# PTV `plan` v1.5 Spec

## Purpose

Bundle five v1.5-tagged beads into one coordinated release:

1. **`ptv-8k9`** — GraphHopper `custom_model` engine for path-strong bike routing
2. **`ptv-53s`** — `--goal commute|day-ride` flag (user-facing surface for the custom model)
3. **`ptv-m7o`** — `--mode bike-only` short-circuit for pure-bike trips
4. **`ptv-dzc`** — `--min-on-path-fraction` hard filter
5. **`ptv-78j`** — elevation surfacing + `--prefer-flat` scoring

Builds on v1.4 (commits through `7bffcc7` on `main`). Existing v1.4 behavior is unchanged when no new flags are passed.

A probe on 2026-05-17 demonstrated that a GraphHopper custom_model request finds dramatically better routes for the "nice ride" use case: Lilydale → Hurstbridge with `strong-prefer-path` model produced a 52 km route with **46 km on dedicated bike paths and zero busy roads** — vs `gh-route`'s `bike_quiet` profile at 56 km with only 25 km on paths. v1.5 makes that capability available via the CLI.

---

## CLI surface (additions)

```
ptv plan <from> <to> [existing flags]
  [--mode <bike-only|bike-train>]      # default: bike-train
  [--goal <commute|day-ride>]          # default: commute
  [--prefer-flat]                      # new soft bias
  [--min-on-path-fraction <f>]         # new hard filter, f in [0,1]
```

`--prefer-bike-path` (v1.3) and `--max-transfers` (v1.2) are unchanged. The new flags compose orthogonally with them.

### Mode semantics

| mode | behavior |
|---|---|
| `bike-train` (default) | Current K=1/K=2 search: bike → train → bike, with optional K=2 transfer |
| `bike-only` | Skip transit search entirely. Single bike leg from origin to destination. Reject if straight-line bike distance > `--max-bike-km` (default 20). |

`--mode bike-only --max-transfers > 0` is rejected with a clear error (no transit, no transfers).

### Goal semantics

| goal | bike-routing engine |
|---|---|
| `commute` (default) | `gh-route --profile bike` (current) |
| `day-ride` | GraphHopper REST `custom_model` request with the path-strong preset (below) |

`--goal day-ride` does NOT change the search algorithm — K=1/K=2 transit search still runs (unless `--mode bike-only`). Only the bike-leg routing is replaced.

### `--prefer-flat` semantics

Adds soft bias to the `recommended` label's cost function (mirrors `--prefer-bike-path`):

```
cost(itinerary) = totalTimeMin
  - PATH_BONUS_PER_KM * bikeKmOnPath        (existing, when --prefer-bike-path)
  - FLAT_BONUS_PER_PERCENT * (100 * flatFraction)   (new, when --prefer-flat)
  + ASCEND_PENALTY_PER_M * totalAscendM             (new, when --prefer-flat)
  + SUSTAINED_CLIMB_PENALTY * (maxSustainedGradePercent * maxSustainedGradeM)  (new, when --prefer-flat)
```

`--prefer-flat` and `--prefer-bike-path` may be combined.

### `--min-on-path-fraction` semantics

Hard filter applied after routing. Drops any itinerary where `bikeKmOnPath / bikeKm < f`. If all itineraries filtered, return near-miss with `constraintsViolated: ['min_on_path_fraction']`.

Validation: `f ∈ [0, 1]`. Reject `--min-on-path-fraction` combined with `--no-enrich` (no path data to filter on).

## Item 1 — GraphHopper custom_model engine (`ptv-8k9`)

### New `external.ts` function

Add to `src/plan/external.ts`:

```ts
type CustomModelPriorityRule = { if: string; multiply_by: number };
type CustomModel = {
  priority: CustomModelPriorityRule[];
  distance_influence: number;
};

const GH_REST_URL = process.env.GH_REST_URL
  ?? 'http://graphhopper.magpie-inconnu.ts.net:8989/route';

export async function ghRouteCustom(
  from: LatLon, to: LatLon, customModel: CustomModel,
): Promise<GhRouteResult | null> {
  const body = {
    points: [[from.lon, from.lat], [to.lon, to.lat]],
    profile: 'bike',
    'ch.disable': true,
    points_encoded: false,
    instructions: false,
    elevation: true,
    details: ['road_class', 'average_slope', 'surface'],
    custom_model: customModel,
  };
  try {
    const r = await fetch(GH_REST_URL, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(body),
    });
    if (!r.ok) return null;
    const data = await r.json() as { paths?: unknown[] };
    const p = data.paths?.[0];
    if (!p) return null;
    return parseGhRoute([{ response: { paths: [p] } }]);
  } catch {
    return null;
  }
}
```

`GhRouteResult` is the existing return type of `ghRouteBike()` plus the new elevation/slope fields from Item 5. Both `ghRouteBike` and `ghRouteCustom` use the same `parseGhRoute()` to extract metrics — they only differ in HOW they obtain the path JSON (CLI subprocess vs REST POST).

### Preset constant

In `src/plan/types.ts`:

```ts
export const DAY_RIDE_CUSTOM_MODEL: CustomModel = {
  priority: [
    { if: 'road_class == SECONDARY',   multiply_by: 0.1 },
    { if: 'road_class == PRIMARY',     multiply_by: 0.05 },
    { if: 'road_class == TRUNK',       multiply_by: 0.05 },
    { if: 'road_class == TERTIARY',    multiply_by: 0.4 },
    { if: 'road_class == RESIDENTIAL', multiply_by: 0.7 },
  ],
  distance_influence: 50,
};
```

These exact values are what the 2026-05-17 probe used to produce the Lilydale → Hurstbridge result. Tunable in a future spec.

### Failure handling

`ghRouteCustom` returns `null` on any error (network, non-2xx, malformed JSON, missing `paths`). The orchestrator treats `null` the same as gh-route failures today: log a warning, fall back to `ghRouteBike()` with the default profile, and continue. The trip still completes; just with reduced quality.

## Item 5 — Elevation surfacing + `--prefer-flat` (`ptv-78j`)

### Parser update

Extend `parseGhRoute()` in `src/plan/external.ts` to extract elevation and slope metrics:

```ts
function parseGhRoute(raw: unknown): {
  km: number; min: number; kmOnPath: number;
  ascendM: number; descendM: number;
  maxSustainedGradePercent: number; maxSustainedGradeM: number;
  flatFraction: number; steepFraction: number;
  geometry: GeoJsonLineString | null;
} | null {
  /* existing distance/time/road_class logic plus: */

  const slope = p.details?.average_slope ?? [] as Array<[number, number, number]>;
  const segDistMeters = (() => {
    const c = p.points?.coordinates ?? [];
    const out: number[] = [];
    for (let i = 0; i < c.length - 1; i++) out.push(haversineM(c[i], c[i+1]));
    return out;
  })();

  let totalD = 0, flatD = 0, steepD = 0;
  let curRun: { endIdx: number; g: number; dist: number } | null = null;
  let hardest: { g: number; dist: number } | null = null;
  for (const [a, b, g] of slope) {
    let d = 0;
    for (let i = a; i < b; i++) d += segDistMeters[i] ?? 0;
    totalD += d;
    const ag = Math.abs(g);
    if (ag < 4) flatD += d;
    if (ag >= 6) steepD += d;
    if (g >= 5) {
      if (curRun && curRun.endIdx === a) {
        curRun.endIdx = b; curRun.dist += d; curRun.g = Math.max(curRun.g, g);
      } else {
        curRun = { endIdx: b, g, dist: d };
      }
      if (!hardest || curRun.dist * curRun.g > hardest.dist * hardest.g) {
        hardest = { g: curRun.g, dist: curRun.dist };
      }
    } else {
      curRun = null;
    }
  }

  return {
    km, min, kmOnPath,
    ascendM: typeof p.ascend === 'number' ? p.ascend : 0,
    descendM: typeof p.descend === 'number' ? p.descend : 0,
    maxSustainedGradePercent: hardest?.g ?? 0,
    maxSustainedGradeM: hardest?.dist ?? 0,
    flatFraction: totalD > 0 ? flatD / totalD : 0,
    steepFraction: totalD > 0 ? steepD / totalD : 0,
    geometry,
  };
}

function haversineM(a: [number, number], b: [number, number]): number {
  const R = 6371000;
  const toRad = (d: number) => (d * Math.PI) / 180;
  const dLat = toRad(b[1] - a[1]);
  const dLon = toRad(b[0] - a[0]);
  const h = Math.sin(dLat/2) ** 2
    + Math.cos(toRad(a[1])) * Math.cos(toRad(b[1])) * Math.sin(dLon/2) ** 2;
  return 2 * R * Math.asin(Math.sqrt(h));
}
```

### Elevation data flow — commute vs day-ride

In `--goal day-ride` mode, the bike-leg result comes from `ghRouteCustom`, which returns all elevation/slope fields directly. The displayed geometry, distance, and elevation are all from the same GraphHopper response — internally consistent.

In `--goal commute` mode (default), the bike-leg distance/duration/geometry comes from `osrm-au` (faster routing engine), and elevation/path data is folded in from the existing `--enrich` (gh-route) call when enabled. This is the same pattern v1.1 established for `bikeKmOnPath` — the elevation metrics describe gh-route's path, which is *similar but not identical* to osrm-au's path. The approximation is acceptable; precision is within ~10% on the same OD pair. When `--no-enrich` is set in commute mode, elevation fields are simply absent.

### Orchestrator aggregation

In both K=1 and K=2 itinerary construction, compute aggregates from the bike legs:

```ts
const ascendM = bikeOut.ascendM + bikeIn.ascendM;
const descendM = bikeOut.descendM + bikeIn.descendM;
const maxSustainedGradePercent = Math.max(
  bikeOut.maxSustainedGradePercent ?? 0, bikeIn.maxSustainedGradePercent ?? 0,
);
const maxSustainedGradeM = bikeOut.maxSustainedGradePercent >= bikeIn.maxSustainedGradePercent
  ? (bikeOut.maxSustainedGradeM ?? 0) : (bikeIn.maxSustainedGradeM ?? 0);
const flatFraction = bikeKm > 0
  ? (bikeOut.km * (bikeOut.flatFraction ?? 0) + bikeIn.km * (bikeIn.flatFraction ?? 0)) / bikeKm
  : 0;
const steepFraction = bikeKm > 0
  ? (bikeOut.km * (bikeOut.steepFraction ?? 0) + bikeIn.km * (bikeIn.steepFraction ?? 0)) / bikeKm
  : 0;
```

For K=2 add the third bike leg if present.

### Score function biases

In `src/plan/score.ts`, add to the cost calculation when `req.preferFlat`:

```ts
function cost(it: Itinerary, req: PlanRequest): number {
  let c = it.totalTimeMin;
  if (req.preferBikePath && typeof it.bikeKmOnPath === 'number') {
    c -= PATH_BONUS_PER_KM * it.bikeKmOnPath;
  }
  if (req.preferFlat) {
    if (typeof it.flatFraction === 'number') c -= FLAT_BONUS_PER_PERCENT * (100 * it.flatFraction);
    if (typeof it.ascendM === 'number') c += ASCEND_PENALTY_PER_M * it.ascendM;
    if (typeof it.maxSustainedGradePercent === 'number' && typeof it.maxSustainedGradeM === 'number') {
      c += SUSTAINED_CLIMB_PENALTY * (it.maxSustainedGradePercent * it.maxSustainedGradeM);
    }
  }
  return c;
}
```

## Commit order

Five commits, each independently revertable. Tests pass after each.

1. **`feat(plan): extract elevation and slope metrics from gh-route`** — `parseGhRoute` enhancement + BikeLeg/Itinerary fields + orchestrator aggregation. Existing behavior unchanged; new fields just populated.

2. **`feat(plan): add ghRouteCustom and --goal day-ride flag`** — GH REST custom_model engine + orchestrator dispatch + DAY_RIDE_CUSTOM_MODEL constant. Without this commit, day-ride falls back to commute behavior.

3. **`feat(plan): add --mode bike-only for direct bike trips`** — orchestrator branch + planBikeOnly helper + CLI flag. Pure short-circuit; doesn't touch transit search.

4. **`feat(plan): add --min-on-path-fraction filter`** — score.ts filter + constraint violation + near-miss + CLI flag.

5. **`feat(plan): add --prefer-flat scoring with elevation penalties`** — score.ts cost function extension + CLI flag.

After commit 5: full suite + integration + e2e all green.

## Out of scope (deferred to v1.6+ / v2)

- `--custom-model-file FILE` — escape hatch for user-supplied custom_model JSON. Useful for power users but not needed for v1.5 since the preset covers the demonstrated case.
- Configurable preset weights (currently the day-ride preset is a hardcoded constant)
- `--goal max-path` third preset — marginal benefit for Lilydale → Hurstbridge; revisit when calibration data on other corridors exists
- `--bike-profile bike_quiet|bike_balanced` — made effectively redundant by `--goal day-ride`; keep gh-route's profiles in `external.ts` but don't expose as a flag
- Popularity overlay (`ptv-5fy`) — separate v2.0 spec
- K=3 transfers — separate spec
- Train pattern polylines on the map — deferred from v1.4

## Risk + mitigation

Bigger spec = more surface for regressions. Mitigations:

- 5 separate commits, each independently revertable.
- New fields are additive only — no JSON schema breaking changes.
- New flags default to current behavior (`--mode bike-train`, `--goal commute`, no `--prefer-flat`, no `--min-on-path-fraction`).
- Custom-model failures fall back to osrm-au; the plan still completes.
- Test counts grow by ~25 unit + 2 integration + 4 e2e.

The probe results (v2 path-strong: 0 km on busy roads vs bike_quiet's 2.3 km) are the calibration anchor. Acceptance: integration test asserts `--goal day-ride` produces `bikeKmOnPath` strictly higher than `--goal commute` for the same Brunswick → Upfield query.

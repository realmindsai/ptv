# PTV `plan` v1.6 Spec

## Purpose

Single-feature release: add `--goal max-path` as a third preset for the GraphHopper `custom_model` engine. The preset uses `distance_influence: 10` and aggressively penalizes residential streets (`0.3`) on top of the existing busy-road penalties — biasing the algorithm toward dedicated bike paths even when the detour is meaningful.

Empirically validated 2026-05-17 across 18 Melbourne corridors. Closes bead **`ptv-uju`**.

Builds on v1.5 (commits through `7dd23b8` on `main`).

---

## Probe results that drove this spec

Re-running 18 cross-line pairs through both presets (full data in `/tmp/melbourne-rides-maxpath.json`):

| corridor | day-ride path-km | max-path path-km | Δ |
|---|---|---|---|
| Camberwell → Lilydale | 23.4 | **43.3** | **+20.0** |
| Hurstbridge → South Morang | 9.7 | 18.1 | +8.5 |
| Hurstbridge → Darebin | 27.9 | **35.9** (98% on path) | +8.0 |
| Box Hill → Lilydale | 22.0 | 29.3 | +7.3 |
| Heidelberg → Clifton Hill | 4.3 | 11.3 | +7.1 |
| Box Hill → Glen Iris | 4.6 | 11.4 | +6.8 |
| Rosanna → Clifton Hill | 5.7 | 12.5 | +6.8 |
| Macleod → Clifton Hill | 8.8 | 13.8 | +5.0 |
| Belgrave → Lilydale | 24.1 | 27.2 | +3.1 |
| Eltham → Reservoir | 14.2 | 16.5 | +2.3 |
| Eltham → Mernda | 17.3 | 19.2 | +1.9 |
| Footscray → Sandringham | 22.6 | 23.3 | +0.6 |
| Williamstown → Brighton Beach | 20.1 | 20.1 | 0.0 (already saturated) |

`max-path` adds 10-40% distance and 5-25% time in exchange for routes with dramatically more on-path mileage, especially in dense-trail corridors. Bay/Maribyrnong corridors were already at ceiling under `day-ride`; no improvement available, no regression.

## CLI surface (additions)

```
ptv plan <from> <to> [existing flags]
  --goal <commute|day-ride|max-path>   # adds 'max-path' as a third value
```

Existing `commute` and `day-ride` semantics are unchanged. `max-path` adds a more aggressive custom_model:

- `distance_influence: 10` (vs day-ride's `50`)
- `residential` multiplier: `0.3` (vs day-ride's `0.7`)
- `tertiary` multiplier: `0.1` (vs day-ride's `0.4`)
- `secondary`/`primary`/`trunk`: `0.02`/`0.01`/`0.01` (vs day-ride's `0.1`/`0.05`/`0.05`)

## Implementation

### New constant in `src/plan/types.ts`

```ts
export const MAX_PATH_CUSTOM_MODEL: CustomModel = {
  priority: [
    { if: 'road_class == SECONDARY',   multiply_by: 0.02 },
    { if: 'road_class == PRIMARY',     multiply_by: 0.01 },
    { if: 'road_class == TRUNK',       multiply_by: 0.01 },
    { if: 'road_class == TERTIARY',    multiply_by: 0.1 },
    { if: 'road_class == RESIDENTIAL', multiply_by: 0.3 },
  ],
  distance_influence: 10,
};
```

The existing `DAY_RIDE_CUSTOM_MODEL` is untouched.

### `PlanGoal` extension

```ts
export type PlanGoal = 'commute' | 'day-ride' | 'max-path';
```

### Orchestrator dispatch (`src/plan/orchestrator.ts`)

The existing `bikeRouteCall(s, from, to)` helper picks the model based on `s.req.goal`:

```ts
async function bikeRouteCall(
  s: SearchState, from: LatLon, to: LatLon,
): Promise<RouteResult> {
  if (s.req.goal === 'day-ride' || s.req.goal === 'max-path') {
    const { DAY_RIDE_CUSTOM_MODEL, MAX_PATH_CUSTOM_MODEL } = await import('./types');
    const model = s.req.goal === 'max-path' ? MAX_PATH_CUSTOM_MODEL : DAY_RIDE_CUSTOM_MODEL;
    const r = await s.deps.external.ghRouteCustom(from, to, model);
    if (r) {
      return { km: r.km, min: r.min, geometry: r.geometry };
    }
  }
  return s.deps.external.osrmRoute('bicycle', from, to);
}
```

The same pattern applies inside `planBikeOnly`.

### CLI validation (`src/commands/plan.ts`)

```ts
if (!['commute', 'day-ride', 'max-path'].includes(opts.goal)) {
  throw new Error(`--goal must be 'commute', 'day-ride', or 'max-path' (got ${opts.goal})`);
}
```

## JSON output changes

None. The schema is unchanged. The `query.goal` field can now be `"max-path"` (previously only `"commute"` or `"day-ride"`).

## CLAUDE.md update

The existing `--goal <commute|day-ride>` synopsis becomes `--goal <commute|day-ride|max-path>`. Add a one-line description for `max-path`:

> **`--goal max-path`** = more aggressive than `day-ride`. Accepts longer routes (often +10-40% distance) to maximize cycleway/path/track mileage. Use when "best ride" is the goal, not "shortest reasonable." Probe: Hurstbridge → Darebin at 98% on path, +20 min vs day-ride.

## Out of scope (deferred)

- `--custom-model-file FILE` — user-supplied custom_model JSON. Power-user escape hatch. Useful but unblocked by `max-path` covering the common case.
- Tunable distance_influence via flag (e.g. `--distance-influence 25` for fine-grained control between day-ride and max-path). Likely v1.7 or v2.0 territory.
- Popularity overlay (`ptv-5fy`)
- K=3 transfers, train pattern polylines on maps
- Re-tuning `day-ride` based on the same probe data (it's already well-calibrated for sparse-trail networks)

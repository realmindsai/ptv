# PTV `plan` v1.3 Spec

## Purpose

Four bounded improvements to v1.2:

1. **bike-path label + `--prefer-bike-path` flag** ‚Äî surface and optionally optimize for cycling on dedicated bike infrastructure.
2. **Hub `stopName` lookup** ‚Äî populate the empty `fromStopName` / `toStopName` on K=2 inner train legs.
3. **Transfer-dwell minimization** ‚Äî penalize long layovers at hubs in the generalized cost so `recommended` prefers tighter connections.
4. **Farther candidate stops** ‚Äî diversify the candidate set so high `--min-bike-km` queries find suitable stops reliably rather than incidentally.

No algorithmic shape changes; no new modules. Builds directly on v1.2 (commits up through `05db1d6` on `main`).

---

## Item 1 ‚Äî bike-path label + flag

### New label

In `src/plan/score.ts`'s `labelAndSort`:

- After deduping/sorting, find the itinerary with the maximum `bikeKmOnPath` among feasible itineraries (those satisfying min/max bike).
- If multiple tie, prefer the one with greater `bikeKm` total.
- If still tied, pick the one already sorted earliest.
- Assign label `'most-bike-path'` to that itinerary.

If no itineraries have `bikeKmOnPath` populated (gh-route unavailable), do not assign the label.

### New ItineraryLabel value

Extend `src/plan/types.ts`:

```ts
export type ItineraryLabel =
  | 'recommended' | 'fastest' | 'most-bike' | 'most-bike-path' | 'fewest-transfers';
```

### New CLI flag

`src/commands/plan.ts`:

```
--prefer-bike-path     Recommend itineraries with more bike-path km
```

When set, the `recommended` label is assigned to the itinerary minimizing a path-biased cost:

```
cost = totalTimeMin - PATH_BONUS_PER_KM * (bikeKmOnPath ?? 0)
```

with `PATH_BONUS_PER_KM = 5` (5 minutes of "value" per km on dedicated paths). The `--prefer-bike-path` flag is independent of `--max-transfers`; it only affects scoring, not search.

When `--prefer-bike-path` is set but no itinerary has `bikeKmOnPath` populated, fall back to the default `recommended` (min `totalTimeMin`) and emit a warning `"--prefer-bike-path requested but no itinerary has gh-route data; using default scoring"`.

### Constant

In `src/plan/types.ts`:

```ts
export const PATH_BONUS_PER_KM = 5;
```

## Item 2 ‚Äî hub stopName lookup

### Problem

K=2 itineraries have empty `legs[1].toStopName` and `legs[2].fromStopName` because the orchestrator only knows the hub `stopId`, not its name. The JSON output today shows `'(hub)'`-equivalent values that consumers must resolve themselves.

### Fix

Extend `src/plan/hubs.ts` to carry names alongside IDs:

```ts
export const HUBS: ReadonlyArray<{ stopId: number; name: string }> = [
  { stopId: 1071, name: 'Flinders Street Station' },
  { stopId: 1181, name: 'Southern Cross Station' },
  /* ...11 more ‚Äî re-run the v1.2 Task 1 search script to fetch
     `actual_name` strings for all 13 stations, then paste them here. */
];

export const HUB_STOP_IDS: number[] = HUBS.map((h) => h.stopId);
const HUB_SET = new Set(HUB_STOP_IDS);
const HUB_NAME_BY_ID = new Map(HUBS.map((h) => [h.stopId, h.name] as const));

export function isHub(stopId: number): boolean {
  return HUB_SET.has(stopId);
}

export function hubName(stopId: number): string {
  return HUB_NAME_BY_ID.get(stopId) ?? '';
}
```

Names come from the PTV API as `<station name> Station`, exactly as Task 1 of v1.2 captured them via the `actual_name` field. The implementer copies those 13 strings from the existing JSON (visible in commit `c33b17f`'s implementer report or by re-running the search script).

In `src/plan/orchestrator.ts` `planK2Hubs`, where the TrainLeg construction sets `toStopName: ''` and `fromStopName: ''`, replace with `hubName(t.hubStopId)`:

```ts
{ mode: 'train', ...
  toStopName: hubName(t.hubStopId),    // was: ''
  ... },
{ mode: 'train', ...
  fromStopName: hubName(t.hubStopId),  // was: ''
  ... },
```

Existing K=1 legs are unaffected.

## Item 3 ‚Äî transfer-dwell minimization

### Problem

Today's Rosanna‚ÜíCaulfield query returned a 28-minute layover at Flinders Street as `recommended`. The score function only optimizes `totalTimeMin`; a tight 7-min connection isn't differentiated from a sloppy 28-min one because both produce the same `totalTimeMin` (the longer wait just shifts the schedule, not the total elapsed).

Actually that's only true for `--arrive-by` queries where `waitMin = 0`. For depart-now queries, longer layover IS reflected in higher `totalTimeMin`. The issue is that "wait" at the hub between train legs isn't currently tracked as a separate field ‚Äî it's absorbed silently into `trainMin` (train1 + train2 sum, with the dwell gap invisible).

Inspect the existing K=2 `trainMin` calculation in `planK2Hubs`:

```ts
const train1Min = (Date.parse(t.arrive1Utc) - Date.parse(t.depart1Utc)) / 60_000;
const train2Min = (Date.parse(t.arrive2Utc) - Date.parse(t.depart2Utc)) / 60_000;
const trainMin = train1Min + train2Min;
```

This sums in-vehicle times only; the hub dwell (`depart2Utc - arrive1Utc`) is NOT included. So today's `totalTimeMin` actually UNDERESTIMATES the trip ‚Äî a 28-min layover doesn't show up at all.

### Fix

1. Add `transferDwellMin` calculation:

```ts
const transferDwellMin = (Date.parse(t.depart2Utc) - Date.parse(t.arrive1Utc)) / 60_000;
```

2. Include it in `totalTimeMin`:

```ts
const totalTimeMin = bikeMin + waitMin + train1Min + transferDwellMin + train2Min;
```

3. Add `transferDwellMin` as a field on `Itinerary` (additive to JSON):

```ts
export type Itinerary = {
  /* ...existing fields... */
  transferDwellMin?: number;  // populated when transfers > 0
};
```

4. Score function uses `totalTimeMin` directly ‚Äî no separate dwell penalty needed, because `totalTimeMin` now includes dwell. The 28-min Rosanna‚ÜíCaulfield example would have its `totalTimeMin` increase from 93 ‚Üí 121, and the 7-min layover alternative would score better.

### No new constant required

The fix is bookkeeping (dwell is real time and should be counted), not a tuned penalty. `TRANSFER_BUFFER_MIN = 5` stays as the minimum allowable transfer time at search time; dwell is whatever the schedule produces, ‚â• 5 min by construction.

## Item 4 ‚Äî farther candidate stops

### Problem

`candidates.ts` currently caps at top-30 by `bikeMin` ascending ‚Äî the 30 closest stops. For queries with high `--min-bike-km`, the closest stops produce bike legs that are too short, so the search either finds nothing or relies on the unintuitive geometry of the train routing to reach a farther stop "by accident." Today's 10-km Rosanna‚ÜíCaulfield query worked partly by luck.

### Fix

Replace the simple top-30 cap with a union of "close" and "far" stops:

```ts
// Sort by bikeMin ascending
const sorted = out.slice().sort((a, b) => a.bikeMin - b.bikeMin);
const TOP_CLOSE = 15;
const TOP_FAR = 15;
const closeIds = new Set(sorted.slice(0, TOP_CLOSE).map((c) => c.stopId));
const farIds = new Set(sorted.slice(-TOP_FAR).map((c) => c.stopId));
const keepIds = new Set([...closeIds, ...farIds]);
const result = out.filter((c) => keepIds.has(c.stopId));
return result;
```

With ‚â§ 30 distinct stops in the union. When the input has ‚â§ 30 stops, all are kept (no truncation occurs).

The 15+15 split is chosen because:
- 15 closest handles the common case (short bike + train + short bike for direct-route queries)
- 15 farthest enables high `--min-bike-km` queries
- The two sets often overlap when the candidate radius is small; dedup keeps the result under 30

### Constants

In `src/plan/types.ts`, replace the single `TOP_N_CANDIDATES` with:

```ts
export const TOP_N_CANDIDATES = 30;        // unchanged: upper bound on total
export const CANDIDATES_CLOSE = 15;        // new
export const CANDIDATES_FAR = 15;          // new
```

`TOP_N_CANDIDATES` becomes documentation-only after this change; keep it for clarity.

## JSON output changes (additive)

- `legs[*].toStopName` and `legs[*].fromStopName` are now populated on K=2 inner train legs (was empty string).
- `itineraries[*].transferDwellMin` field appears when `transfers > 0` (new optional field).
- `itineraries[*].totalTimeMin` for K=2 trips is now larger than before by the dwell amount ‚Äî this is a correctness fix, not a regression. Existing consumers that compared `totalTimeMin` against external times (e.g. wall-clock arrival) will become more accurate.
- `itineraries[*].labels` may include the new `'most-bike-path'` value.
- `warnings` may include the new `'--prefer-bike-path requested but no itinerary has gh-route data...'` message.

No fields are removed or renamed.
